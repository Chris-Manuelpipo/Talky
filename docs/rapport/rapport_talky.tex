\chapter*{Résumé}
\addcontentsline{toc}{chapter}{Résumé}

Ce rapport présente la conception et l'implémentation de \textbf{Talky}, une application mobile de messagerie instantanée et d'appels audio/vidéo temps réel, développée dans le cadre du module \textit{Projet Base de Données} à l'École Nationale Supérieure Polytechnique de Yaoundé.

Talky cible les usages professionnels internes à l'entreprise et offre, à l'instar de WhatsApp, un ensemble riche de fonctionnalités : messagerie texte et multimédia, appels individuels et de groupe via WebRTC, publication de statuts éphémères de 24 heures, gestion des contacts, historique des appels et personnalisation de l'interface.

L'application repose sur une architecture client-serveur hybride : \textbf{Flutter + Riverpod} côté client mobile, \textbf{Firebase} (Firestore, FCM, Realtime Database) pour la persistance et les notifications, \textbf{Cloudinary} pour le stockage des médias, et un serveur de signalisation \textbf{Node.js/Socket.IO} déployé sur Render.com pour la coordination des appels WebRTC.

Le document couvre l'ensemble des phases du cycle de développement : analyse du problème, étude de l'existant, modélisation des données (MCD, MLD), modélisation UML, architecture technique, implémentation et perspectives.

\medskip
\textbf{Mots-clés :} messagerie instantanée, WebRTC, Flutter, Firebase, Firestore, Socket.IO, Cloudinary, temps réel, NoSQL, application mobile.

\chapter{Introduction générale}

\section{Contexte}

La communication numérique a radicalement transformé la manière dont les individus et les organisations échangent de l'information. Les applications de messagerie instantanée --- WhatsApp, Telegram, Signal, Microsoft Teams --- sont désormais au cœur des pratiques professionnelles et personnelles à l'échelle mondiale. En Afrique subsaharienne, et particulièrement au Cameroun, la pénétration des smartphones et l'essor des réseaux mobiles 4G ont créé un besoin croissant de solutions de communication rapides, fiables et économiques.

Dans le cadre du module \textit{Projet Base de Données} du Département de Génie Informatique de l'ENSPY, notre groupe a été chargé de concevoir et d'implémenter une solution logicielle complète répondant à des besoins réels de communication. Nous avons choisi de développer \textbf{Talky}, une application de messagerie instantanée et d'appels temps réel pensée pour un usage professionnel interne.

\section{Problématique}

Les entreprises et organisations camerounaises utilisent majoritairement des solutions propriétaires étrangères pour leurs communications internes. Ces solutions présentent plusieurs inconvénients majeurs :

\begin{itemize}[leftmargin=1.5cm]
  \item \textbf{Dépendance aux serveurs étrangers :} les données des employés transitent par des infrastructures hors du contrôle de l'organisation.
  \item \textbf{Absence de personnalisation :} impossibilité d'adapter l'outil aux processus internes ou à l'identité visuelle de l'entreprise.
  \item \textbf{Coûts de licence :} les solutions professionnelles comme Teams ou Zoom impliquent des abonnements onéreux.
  \item \textbf{Problèmes de confidentialité :} la collecte de métadonnées par les grandes plateformes représente un risque pour la confidentialité des échanges.
\end{itemize}

\begin{tcolorbox}[colback=ENSPYgray,colframe=ENSPYblue,title={\bfseries\color{white}Question centrale}]
\itshape Comment concevoir et implémenter une application mobile de communication interne à la fois performante, sécurisée, économique et adaptable, tout en s'appuyant sur des technologies modernes accessibles à une équipe étudiante~?
\end{tcolorbox}

\section{Objectifs du projet}

\subsection{Objectif général}
Concevoir et développer une application mobile multiplateforme de messagerie instantanée et d'appels audio/vidéo temps réel, destinée à un usage professionnel interne.

\subsection{Objectifs spécifiques}
\begin{enumerate}[leftmargin=1.5cm]
  \item Implémenter un système de messagerie textuelle et multimédia avec accusés de réception.
  \item Intégrer des appels audio et vidéo individuels et de groupe via WebRTC.
  \item Développer un système de statuts éphémères (24 heures) similaire aux Stories.
  \item Assurer les notifications push en temps réel via Firebase Cloud Messaging.
  \item Proposer une interface utilisateur intuitive, moderne et accessible.
  \item Mettre en place une authentification robuste par OTP.
\end{enumerate}

\section{Structure du rapport}

Le rapport s'articule en sept chapitres. Le chapitre~2 présente l'étude de l'existant. Le chapitre~3 couvre la modélisation des données (MCD/MLD). Le chapitre~4 présente la modélisation UML. Le chapitre~5 décrit l'architecture technique. Le chapitre~6 détaille l'implémentation. Le chapitre~7 conclut et ouvre des perspectives.

\chapter{Analyse et étude de l'existant}

\section{Panorama des solutions existantes}

\subsection{WhatsApp}
WhatsApp Messenger, propriété de Meta, est l'application de messagerie la plus utilisée au monde avec plus de 2,7 milliards d'utilisateurs actifs. Elle offre messagerie texte et multimédia, appels audio/vidéo, statuts de 24 heures et chiffrement de bout en bout. Son principal inconvénient pour un usage professionnel reste l'absence de contrôle sur les données et d'impossibilité de personnalisation.

\subsection{Telegram}
Telegram se distingue par ses groupes de grande capacité (jusqu'à 200\,000 membres), ses canaux de diffusion, la prise en charge de bots et une API ouverte. Il propose également un mode Secret avec chiffrement de bout en bout et auto-destruction des messages.

\subsection{Signal}
Signal est la référence en matière de confidentialité, avec un chiffrement de bout en bout basé sur son propre protocole. Il reste toutefois limité en fonctionnalités professionnelles avancées.

\subsection{Microsoft Teams / Slack}
Ces solutions professionnelles offrent des fonctionnalités de collaboration avancées (partage de fichiers, réunions, intégrations tierces) mais impliquent des coûts significatifs et une infrastructure complexe.

\section{Positionnement de Talky}

\begin{infobox}[Valeur ajoutée de Talky]
\begin{itemize}
  \item \textbf{Maîtrise des données :} l'organisation déploie son propre serveur de signalisation et contrôle intégralement la base Firebase.
  \item \textbf{Mode fantôme (Ghost Mode) :} l'utilisateur peut masquer son statut de connexion sans quitter l'application.
  \item \textbf{Messages confidentiels :} auto-destruction configurable de 5 secondes à une durée personnalisée (fonctionnalité planifiée).
  \item \textbf{Traduction instantanée intégrée :} prévu pour la version 2.0.
  \item \textbf{Architecture extensible :} base Flutter multiplateforme (Android, iOS, Web).
\end{itemize}
\end{infobox}

\section{Analyse des besoins}

\subsection{Besoins fonctionnels}

\begin{enumerate}[leftmargin=1.5cm,label=\textbf{BF\arabic*}]
  \item Inscription via numéro de téléphone avec vérification OTP.
  \item Envoi/réception de messages texte, images, vidéos, audios et fichiers.
  \item Appels audio et vidéo individuels et de groupe.
  \item Publication de statuts éphémères visibles 24 heures.
  \item Création et gestion de groupes de discussion.
  \item Notifications push même lorsque l'application est fermée.
  \item Archivage, épinglage et suppression de conversations.
  \item Affichage du statut de présence des contacts.
\end{enumerate}

\subsection{Besoins non fonctionnels}

\begin{enumerate}[leftmargin=1.5cm,label=\textbf{BNF\arabic*}]
  \item \textbf{Performance :} latence des messages inférieure à 500 ms.
  \item \textbf{Disponibilité :} serveur opérationnel 24h/24, 7j/7.
  \item \textbf{Sécurité :} tokens et clés API non exposés dans le code source.
  \item \textbf{Scalabilité :} groupes jusqu'à 256 membres.
  \item \textbf{Maintenabilité :} architecture modulaire par fonctionnalités.
  \item \textbf{Compatibilité :} Android API 21+ (Android 5.0 minimum).
\end{enumerate}

\chapter{Modélisation des données}

\section{Présentation du système de stockage}

Talky repose sur une architecture de données \textbf{NoSQL distribuée} combinant Firebase Firestore (persistance principale) et Firebase Realtime Database (présence temps réel).

\begin{warningbox}[Justification du choix NoSQL]
Le recours à Firestore plutôt qu'un SGBD relationnel est motivé par~: (1)~la synchronisation temps réel sans polling, (2)~la flexibilité du schéma pour les messages hétérogènes, (3)~l'intégration native avec Firebase Auth et FCM, et (4)~la scalabilité horizontale automatique.
\end{warningbox}

\section{Modélisation mathématique}

\subsection{Complexité des requêtes Firestore}

Soit $C$ le nombre total de conversations et $k$ le nombre de résultats retournés.
Avec l'index composite, la complexité d'une requête de listing est :
\[
  T_{\text{query}} = O(\log C + k)
\]
Sans index, elle serait $O(C)$ (scan complet), ce qui deviendrait inacceptable à grande échelle.

\subsection{Modèle d'expiration des statuts}

Un statut $s$ est \emph{valide} à l'instant $t$ (en secondes Unix) si et seulement si :
\[
  t_{\text{created}}(s) \;\leq\; t \;\leq\; t_{\text{created}}(s) + 86400
\]

Le taux de vues d'un statut est :
\[
  \tau_{\text{view}}(s) = \frac{|\,viewedBy(s)\,|}{|\,contacts(u)\,|} \times 100 \quad (\%)
\]

\subsection{Latence d'un appel WebRTC}

La latence totale perçue est la somme de trois composantes :
\[
  L_{\text{total}} = L_{\text{signaling}} + L_{\text{ICE}} + L_{\text{media}}
\]
\begin{itemize}[noitemsep]
  \item $L_{\text{signaling}} \approx 50$--$200$ ms (Socket.IO sur Render.com)
  \item $L_{\text{ICE}} \approx 100$--$500$ ms (négociation STUN/TURN)
  \item $L_{\text{media}} < 150$ ms (recommandation ITU-T G.114 pour la voix)
\end{itemize}

\subsection{Présence en temps réel --- Firebase Realtime Database}

La structure JSON utilisée pour la présence est :
\begin{lstlisting}[language=Java,caption={Structure Realtime Database pour la présence}]
{
  "presence": {
    "{userId}": {
      "isOnline": true,
      "lastSeen": 1704067200000   // timestamp Unix ms
    }
  }
}
\end{lstlisting}

\section{Architecture Flutter --- Feature-First}

\begin{infobox}[Structure modulaire du projet Flutter]
\texttt{lib/}
\begin{itemize}[noitemsep]
  \item \texttt{core/} --- Services transverses : notifications, présence, routage, thème, cache
  \item \texttt{features/auth/} --- Inscription, OTP, gestion de session Firebase
  \item \texttt{features/chat/} --- Messagerie, médias, conversations, groupes
  \item \texttt{features/calls/} --- Appels WebRTC audio/vidéo (individuel et groupe)
  \item \texttt{features/status/} --- Statuts éphémères 24h
  \item \texttt{features/settings/} --- Paramètres, thème, langue, profil
  \item \texttt{shared/} --- Widgets réutilisables (\texttt{TalkyButton}, \texttt{TalkyTextField})
\end{itemize}
\end{infobox}

Chaque fonctionnalité est découpée en trois sous-couches~:
\begin{enumerate}
  \item \texttt{domain/} --- Modèles de données purs (classes Dart sans dépendances Firebase).
  \item \texttt{data/} --- Services (accès Firestore, Socket.IO) et providers Riverpod.
  \item \texttt{presentation/} --- Écrans et widgets Flutter.
\end{enumerate}

\section{Système de notifications push}

\begin{warningbox}[Point clé --- Notifications data-only]
Les notifications FCM de Talky sont envoyées en mode \textbf{data-only} (sans champ \texttt{notification} dans le payload). Ce choix est fondamental~: sur Android, la présence d'un champ \texttt{notification} entraîne la prise en main par le système Android, contournant Flutter et empêchant la navigation vers l'écran d'appel entrant. Avec un payload \textit{data-only}, Flutter reçoit le message via \texttt{onMessage}/\texttt{onMessageOpenedApp} et contrôle lui-même l'affichage via \texttt{flutter\_local\_notifications}.
\end{warningbox}

\section{Fonctionnalités implémentées}

\subsection{Authentification par OTP}

L'authentification utilise Firebase Phone Authentication. L'utilisateur saisit son numéro international, reçoit un SMS de vérification, entre le code à 6 chiffres, puis complète son profil (nom, photo). Le routage est géré par GoRouter avec redirection automatique selon l'état d'authentification et la complétude du profil Firestore.

\subsection{Messagerie instantanée}

La messagerie est implémentée avec Firestore en temps réel (listeners \texttt{snapshots()}). Les fonctionnalités comprennent~:
\begin{itemize}
  \item Messages texte, images, vidéos, audio et fichiers.
  \item Réponse contextuelle à un message avec prévisualisation (reply).
  \item Modification et suppression de messages (pour tous ou pour soi).
  \item Accusés de réception : envoyé $\to$ reçu $\to$ lu (double coche bleue).
  \item Épinglage et archivage de conversations.
  \item Groupes jusqu'à 256 membres.
\end{itemize}

\subsection{Appels audio/vidéo --- WebRTC}

Les appels P2P utilisent le protocole WebRTC avec le serveur STUN/TURN        \texttt{openrelay.metered.ca}. La signalisation est gérée via Socket.IO. Les appels de groupe utilisent une architecture \textit{mesh} complète.

\begin{lstlisting}[language=Java,caption={Création de la RTCPeerConnection --- call\_service.dart}]
Future<void> _createPeerConnection() async {
  final config = {
    'iceServers': [
      {'urls': 'stun:openrelay.metered.ca:80'},
      {
        'urls':       'turn:openrelay.metered.ca:80',
        'username':   'openrelayproject',
        'credential': 'openrelayproject',
      },
    ],
  };
  _peerConnection = await createPeerConnection(config);
  _peerConnection!.onIceCandidate = (candidate) {
    _socket!.emit('ice_candidate', {
      'targetUserId': _remoteUserId,
      'candidate':    candidate.toMap(),
    });
  };
}
\end{lstlisting}

\subsection{Statuts éphémères}

Les statuts (texte avec couleur de fond, image, vidéo) expirent automatiquement après 24 heures. Le système enregistre les vues et les likes. Une requête Firestore filtre les documents dont \texttt{expiresAt > now()}.

\subsection{Gestion de la présence}

La présence en temps réel utilise Firebase Realtime Database avec la fonctionnalité \texttt{onDisconnect()} pour mettre à jour automatiquement le statut hors-ligne lors d'une déconnexion, y compris inopinée. Le \textbf{Ghost Mode} permet de masquer sa présence.

\chapter{Conclusion et perspectives}

\section{Bilan}

Ce projet nous a permis de concevoir et d'implémenter de A à Z une application de messagerie professionnelle en mobilisant un large spectre de technologies modernes. Les principaux livrables sont~:

\begin{itemize}
  \item Une application Android fonctionnelle avec messagerie texte et multimédia complète.
  \item Un système d'appels audio/vidéo P2P via WebRTC avec prise en charge des appels de groupe.
  \item Un système de statuts éphémères 24h avec compteur de vues et réactions.
  \item Un serveur de signalisation Node.js déployé en production sur Render.com.
  \item Une architecture NoSQL bien structurée avec index Firestore optimisés.
  \item Un système de notifications push robuste basé sur FCM data-only.
\end{itemize}

Sur le plan académique, ce projet a renforcé nos compétences en modélisation (MCD/MLD/UML), architecture logicielle, développement mobile multiplateforme et gestion de bases de données NoSQL temps réel.

\section{Difficultés rencontrées}

\begin{enumerate}
  \item \textbf{Notifications FCM sur Android~:} la présence du champ \texttt{notification} dans le payload FCM provoquait l'interception par Android et rendait inopérante la navigation vers l'écran d'appel. La solution~: messages \textit{data-only}.
  \item \textbf{Écran d'appel entrant~:} l'utilisation de \texttt{FlutterActivity} comme parent de l'activité Android native générait un second moteur Flutter isolé. Le passage à \texttt{AppCompatActivity} a résolu ce problème.
  \item \textbf{Candidats ICE prématurés~:} des candidats ICE étaient émis avant que la connexion peer soit prête, causant des échecs silencieux.
  \item \textbf{Contraintes Firebase Spark~:} Firebase Storage étant limité géographiquement, nous avons adopté Cloudinary comme alternative.
\end{enumerate}

\section{Perspectives d'évolution}

\begin{enumerate}[leftmargin=1.5cm,label=\textbf{P\arabic*}]
  \item \textbf{Traduction instantanée~:} intégration de l'API Google Translate pour traduire les messages dans la langue de l'utilisateur.
  \item \textbf{Mode confidentiel~:} messages à auto-destruction avec durée configurable et prévention des captures d'écran.
  \item \textbf{Version web~:} portage Flutter Web ou React.js partageant la même infrastructure Firebase.
  \item \textbf{Chiffrement de bout en bout~:} implémentation du protocole Signal avant stockage Firestore.
  \item \textbf{Déploiement iOS~:} portage et certification App Store.
  \item \textbf{Tableau de bord d'administration~:} interface web de gestion et monitoring.
\end{enumerate}

En définitive, \textbf{Talky} constitue une démonstration concrète qu'une équipe académique peut, avec les outils modernes disponibles, concevoir et livrer une solution de communication complète et opérationnelle. Ce projet pose les fondations d'un produit qui, avec les évolutions planifiées, pourrait être déployé à l'échelle d'une organisation.

    },
  };

});

\chapter{Règles de sécurité Firestore}
\begin{lstlisting}[language=Java,caption={Règles Firestore recommandées}]
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Profils : lecture publique, ecriture par le proprietaire
    match /users/{userId} {
      allow read:  if request.auth != null;
      allow write: if request.auth.uid == userId;
    }

    // Conversations : acces aux participants uniquement
    match /conversations/{convId} {
      allow read, write: if request.auth.uid in
        resource.data.participantIds;

      match /messages/{msgId} {
        allow read, write: if request.auth.uid in
          get(/databases/$(database)/documents/
              conversations/$(convId))
            .data.participantIds;
      }
    }

    // Statuts : lecture authentifiee, ecriture par auteur
    match /statuses/{statusId} {
      allow read:  if request.auth != null;
      allow write: if request.auth.uid
                   == resource.data.userId;
    }
  }
}
\end{lstlisting}
